# 프로젝트 Context

1. 개요
    - 25명의 미디어디자인 수업에서 진행할 실습을 위한 플랫폼입니다
    - 학생들은 28일간 매일 토론을 진행합니다 (3/31~4/28)
    - 역할 개요
        1. 아젠다 세터 : 토론 주제를 선정합니다
        2. 아키텍트 : 토론 인터페이스를 설계합니다
        3. 참여자 : 토론에 참여합니다
2. 진행방식
    - 토론장의 인터페이스, 토론 주제는 매일 바뀝니다
        - 매일 오후 6시에 시작하고 다음 날 오후 6시에 종료됩니다
    - 25명의 아젠다 세터가 매일 돌아가며 주제를 제안합니다
    - 25명의 아키텍트는 매일 돌아가며 토론 인터페이스를 설계합니다
    - 아젠다 세터와 아키텍트는 자신이 만든 토론에는 참여하지 않습니다
3. 역할
    - 아젠다 세터와 아키텍트는 서로 상의하지 않습니다
    1. 아젠다 세터
        - 목표 : 최대한 격한 토론을 만들어 낸다
        - KPI : 찬성:반대 비율이 50%에 가까움 수치 * 토론 체류시간
        - 주제 선정 조건: 찬성과 반대가 명확히 갈려야 하며, "둘 다 좋다"는 절충안이 나오기 어려운 주제여야 합니다. 따라서 취향에 대한 이야기보다 가치판단에 대한 주제여야 합니다.
        - 주제 제출
            - 토론 시작 전날 오후 6시까지 그레고리에게 DM으로 제출
            - 주제 + 주제 선정에 대한 가설
        - 다음날 토론 리뷰 시에 아래 내용 작성
            - 토론 주제에 선정에 대한 가설
            - 가설에 대한 레슨런
    2. 아키텍트
        - 목표 : 서로 설득당할 수 있는 토론을 만들어 낸다
        - KPI : 찬/반이 서로 설득당하는 비율 * log(체류시간(difficulty))
        - 산출물 조건 :
            - URL이 있는 웹사이트여야 합니다
            - 버그가 발생해서 제대로 집계되지 않은 것은 본인의 책임입니다. 따라서 시작하기 전에 충분히 완성되었는지 검토할 책임이 있습니다.
            - 6시 정각에 URL 접속 불가 시, 즉시 그레고리의 기본 UI로 대체하며 해당 아키텍트는 당일 점수 0점 처리됩니다.
            - 자기가 제출하기 이전에 다른 사람들이 적용했던 인터페이스를 적용하는 것에 제한을 두지 않습니다.
            - 그래픽 요소는 반드시 필요와 의도에 의해서 컨트롤합니다. 단순히 꾸미기 위한 디자인은 엄격하게 금지합니다.
        - 웹사이트 제출
            - 토론 시작 전날 오후 6시까지 그레고리에게 url으로 제출
            - 인터페이스 설계에 대한 가설
        - 다음날 토론 리뷰 시에 아래 내용 작성
            - 인터페이스 설계에 대한 가설
            - 가설에 대한 레슨런
    3. 참여자
        - 매일 할 일
            1. 오늘 진행되는 토론 참여
            2. 전날 진행되었던 토론 리뷰
        - 토론 참여: 하루에 한 번 토론에 최소 한 번 이상 참여해야 합니다. 여러 번 참여해도 상관없으며 참여 횟수에는 제한을 두지 않습니다.
        - 토론 리뷰: 전날 참여했던 토론에 대해 인터페이스 분석 글을 작성해야 합니다. 작성은 오후 6시에 닫힙니다. (분석 글 작성 가이드) 이번에 바뀐 UI에 의해서 토론의 양상이 어떻게 달라졌는가?
            - 그레고리가 매일 베스트 인사이트를 선정합니다.
4. 스케줄링
    - 첫 3일간(3/31~4/2)은 그레고리가 아젠다 세터 + 아키텍트를 모두 담당. 학생 25명 전원 참여자
    - 4일차(4/3)부터 학생이 돌아감. 매일 학생 1명이 아젠다 세터, 다른 1명이 아키텍트
    - 각 학생은 28일간 아젠다 세터 1회 + 아키텍트 1회를 서로 다른 날에 수행
    - 자기가 아젠다 세터 또는 아키텍트인 날에는 참여자로 참여하지 않음 (read-only)
    - 첫 3일간 아젠다 세터, 아키텍트를 자원하는 학생은 가산점이 있습니다.
    - 학생들과 일자별로 랜덤으로 스케줄링 합니다.
    - 스케줄 조정이 필요한 경우 자유롭게 협의해서 조정하고 그레고리에게 알려줍니다.
5. 평가기준
    1. 참여점수 : 매일 참여자로써 참여하기, 아키텍트, 아젠다세터를 모두 수행해야 참여점수가 부여됩니다
    2. 평가점수 : 
        1. 리포트
            1. 데일리 리포트
                1. 참여자 : 코멘트에서 인사이트를 잘 뽑아냈는가
                2. 아젠다 세터, 아키텍트 : 가설-레슨런 작성
            2. 파이널 리포트 : 변화하는 UI에 의해서 토론의 양상이 어떻게 달라졌는가?
        2. 가산점
            1. 아젠다 세터 가산점 (아젠다 세터 참조)
            2. 아키텍트 가산점 (아키텍트 참조)
            3. 첫 3일간 아젠다 세터, 아키텍트를 하는 사람 가산점




# 토론 플랫폼

1. 접속

- 사용자는 총 26명
- 그레고리 : 관리자
- 25명 학생들은 닉네임이 각자 있고, 닉네임+학번을 입력해야 접속할 수 있음
- 한 번 접속하면 localstorage를 통해서 접속한 상태를 유지

2. 리스트 뷰 (메인 화면)

- 학생이 매일 토론에 참여하고 리뷰를 작성하는 메인 화면
- 여러 토론들을 리스트로 모아 볼 수 있다.
- 토론들은 최신 순으로 나열된다
- 토론 아이템을 클릭하면 상태에 따라 다른 동작 (아래 토론 아이템 섹션 참조)
- 28일 테이블(캘린더)과의 역할 구분:
    - 리스트 뷰: 토론 참여 + 찬반 투표 + 코멘트 작성 (행동하는 곳)
    - 28일 테이블: 전체 스케줄 확인 + 관리 (보는 곳)

3. 토론 아이템

- 메타데이터 : url, title, agendasetter, architect, starttime, 평균체류시간, 찬성수, 반대수, 코멘트
- starttime : 연/월/일. 그 날 오후 6시부터 토론이 activate 되고 다음날 오후 6시에 deactivate 된다
- 대기중 / 진행중 / 집계중 / 종료 스테이터스가 있다
- 상태 전환은 서버 기반으로 자동 전환된다 (Cloud Functions 스케줄 트리거)
    - 대기중 → 진행중 : starttime 도래 시 (해당 날짜 오후 6시)
    - 진행중 → 집계중 : starttime 다음 날 오후 6시
    - 집계중 → 종료 : 집계 시작 후 24시간 경과 시 (starttime + 2일 오후 6시)
    - 서버 시간 기준으로 전환하므로 클라이언트 시간 차이에 영향받지 않음
    1. 대기중 : starttime이 당도하지 않아 아직 activated 되지 않은 토론.
        1. 리스트 표시 데이터: 날짜, 스테이터스, title, agendasetter, architect
    2. 진행중 : 현재 active한 토론
        1. 학생들은 진행 중인 토론에 대해서 찬성/반대 중 하나를 입력할 수 있다. 최초 한 번만 입력 가능. 그 후에는 수정 불가.
        2. 리스트 표시 데이터 : 날짜, 스테이터스, title, agendasetter, architect
        3. 토론을 클릭하면:
            1. 찬/반 미입력 시 → 토론 플랫폼 내에서 찬/반 선택 화면 표시
            2. 찬/반 선택 완료 후 → 해당 url로 이동 (닉네임 + 찬반 파라미터 포함)
            3. 이미 찬/반을 입력한 경우 → 바로 해당 url로 이동
    3. 집계중 : 어제 active했던 토론. 24시간동안 집계를 한다.
        1. 리스트 표시 데이터 : 날짜, 스테이터스, title, agendasetter, architect, 찬성수, 반대수, 평균체류시간, 코멘트 작성 페이지로 넘어가는 버튼
        2. 토론을 클릭하면 해당 url로 넘어감
        3. 코멘트 작성 페이지
            1. 참여자는 집계 중 시간 동안 다음 데이터를 입력할 수 있다: 상대 진영에 설득되었는지 여부 y/n, 토론분석(500자 이내의 긴 텍스트)
            2. 코멘트 작성 후에 원하면 수정할 수 있다. 단 집계중 시간 이내에.
            3. 아젠다 세터와 아키텍트는 집계 중 시간 동안 다음 데이터를 입력할 수 있다: 가설, 토론분석 (500자 이내의 긴 텍스트)
        4. 집계 중에는 다른 사람의 코멘트를 볼 수 없다.
    4. 종료 : 집계가 끝난 토론 (집계중 24시간 경과 후 자동 전환)
        1. 리스트 표시 데이터 : 날짜, 스테이터스, title, agendasetter, architect, 찬성수, 반대수, 평균체류시간, 코멘트 리스트를 볼 수 있는 버튼, 베스트인사이트 닉네임 리스트
        2. 토론을 클릭하면 해당 url로 넘어감
        3. 코멘트 리스트 : 닉네임+코멘트
            1. 그레고리는 특정 코멘트에 ‘베스트 인사이트’ 버튼을 누를 수 있다
            2. 버튼을 누르면 코멘트 입력란이 나오고, 입력하면 ‘베스트 인사이트’ 표시와 함께 선정 이유가 표시된다.
            3. ‘베스트 인사이트’는 상단에 고정된다.
            4. 베스트 인사이트는 여러 개가 될 수 있다. 
- 토론을 클릭하면 해당 url로 넘어가되, 학생의 닉네임과 찬/반 여부를 같이 파라메터로 넘긴다


4. 28일 테이블 (캘린더)

- 28일간의 모든 토론을 하나의 테이블로 표시
- 학생, 그레고리 모두 볼 수 있음. 단, 수정은 그레고리만 가능
- 오늘 날짜가 하이라이트 표시됨
- 상태(status)는 서버 기반으로 자동 전환되며, 색깔로 구분

- 테이블 컬럼:
    | 날짜 | status | 아젠다세터 | 아키텍트 | title | url | 찬성 | 반대 | 체류시간 | 코멘트수 | 미참여 |
    - status: 대기중 / 진행중 / 집계중 / 종료
    - 찬성, 반대, 체류시간, 코멘트수, 미참여: 집계중/종료 상태에서만 표시

- 학생 뷰:
    - 테이블 read-only
    - 자기 이름이 들어간 행 하이라이트 (내가 아젠다세터/아키텍트인 날)
    - 전체 스케줄을 미리 확인할 수 있음

- 그레고리 뷰 (수정 모드):
    - 셀 클릭으로 인라인 수정 (아젠다세터, 아키텍트, url, title, starttime)
    - 빈 셀은 시각적으로 강조 (아직 미입력)
    - 행 추가/삭제 가능
    - csv 다운로드 버튼

- 행 클릭 → 상세 모달:
    - 코멘트 리스트 (닉네임 + 코멘트)
    - 베스트 인사이트 선정 (그레고리 전용): 특정 코멘트에 선정 버튼 + 선정 이유 입력
    - 베스트 인사이트는 상단 고정, 여러 개 가능
    - 미참여자 목록, 코멘트 미입력자 목록
    - 평균 체류시간 수동 입력/수정 (그레고리 전용)



# 기술 스택

| 항목 | 기술 스택 | 이유 |
|------|----------|------|
| 토론 플랫폼 | Next.js + Firestore | 그레고리만 관리. 학생은 이 코드를 보지 않음 |
| Core Script | 순수 JS (의존성 없음) | 학생의 어떤 프로젝트에든 삽입 가능해야 함 |
| 기본 템플릿 | 순수 HTML/CSS/JS | 학생 진입장벽 최소화. 단일 폴더로 배포 가능 |
| 상태 자동 전환 | Firebase Cloud Functions (스케줄 트리거) | 서버 시간 기준 전환, 클라이언트 의존성 없음 |
| Core Script 호스팅 | TBD (Firebase Hosting, Vercel 등) | 안정적 CDN 필요 |

- 토론 플랫폼과 기본 템플릿은 별개의 프로젝트
- 토론 플랫폼의 기술 선택이 학생에게 영향을 주지 않음
- 기본 템플릿을 순수 HTML로 제공하되, 학생이 원하면 React, Svelte 등 자유롭게 전환 가능 (core script만 삽입하면 됨)


# Firestore 데이터 모델

```
users/{nickname}
    - nickname: string
    - studentId: string (학번)
    - role: "admin" | "student"

debates/{debateId}
    - title: string
    - url: string
    - agendaSetter: string (닉네임)
    - architect: string (닉네임)
    - startTime: timestamp (해당 날짜 오후 6시)
    - status: "pending" | "active" | "reviewing" | "closed"
    - stats:
        - proCount: number
        - conCount: number
        - avgDuration: number (초)
        - commentCount: number

debates/{debateId}/votes/{nickname}
    - side: "pro" | "con"
    - votedAt: timestamp

debates/{debateId}/comments/{nickname}
    - role: "participant" | "architect" | "agendasetter"
    - persuaded: boolean (참여자만, 설득 여부)
    - hypothesis: string (아젠다세터/아키텍트만, 가설)
    - analysis: string (토론분석, 500자 이내)
    - isBestInsight: boolean
    - bestInsightReason: string (그레고리가 작성)
    - createdAt: timestamp
    - updatedAt: timestamp

debates/{debateId}/sessions/{nickname}
    - lastHeartbeat: timestamp
    - totalDuration: number (초, heartbeat 기반 누적)
    - isActive: boolean

debates/{debateId}/payloads/{nickname}
    - data: map (custom_payload, 아키텍트가 자유 설계)
    - updatedAt: timestamp
```

- Firestore 보안 규칙:
    - debates 컬렉션: admin만 write, 학생은 read only
    - votes: status가 "active"인 토론에만 write 가능, 본인 닉네임 문서만
    - comments: status가 "reviewing"인 토론에만 write 가능, 본인 닉네임 문서만
    - sessions/payloads: status가 "active"인 토론에만 write 가능 (core script 우회 방지)
    - bestInsight 관련 필드: admin만 write


# 기본 토론 플랫폼

1. 개요

- 그레고리가 기본 토론 웹사이트(템플릿)를 만들어 제공한다
- 학생(아키텍트)은 이 템플릿을 기반으로 자신만의 토론 인터페이스를 만든다
- 학생 닉네임과 찬/반 여부는 토론 플랫폼에서 URL 파라메터로 넘어온다
    - 파라메터로 넘어오지 않는 경우는 토론에 참여할 수 없다
- 아젠다 세터, 아키텍트인 경우 토론에 참여할 수 없다. 하지만 read-only 로 볼 수만 있다.

2. 기본 템플릿

- 그레고리가 만든, 그대로 동작하는 완성된 토론 웹사이트
- 제공 방식: GitHub 레포 clone 또는 파일 직접 배포
- 역할:
    - 학생이 처음부터 만들 필요 없이, 동작하는 예시를 보고 시작할 수 있게 함
    - core script 연동이 이미 되어 있어서, 바로 배포하면 기본 토론 플랫폼으로 동작
    - 첫 3일간(그레고리 담당 기간)에는 이 기본 템플릿이 그대로 사용됨
- 기본 템플릿에 포함된 것:
    - core script 태그 (이미 삽입됨)
    - 기본 토론 UI (찬/반 표시, 닉네임 표시, 기본 레이아웃)
    - window.DebateCore 연동 예시 코드
- 학생의 작업 흐름:
    1. 기본 템플릿을 받는다 (clone 또는 파일 다운로드)
    2. 자기 호스팅에 배포한다 (Vercel, Netlify 등)
    3. data-architect-id를 본인 닉네임으로 변경한다
    4. 정상 동작 확인 후 UI를 자유롭게 수정한다
- 학생은 템플릿을 완전히 뜯어고쳐도 되고, 부분만 수정해도 됨
- 극단적으로 템플릿을 버리고 처음부터 만들어도 됨 (core script만 삽입하면 프레임워크 자유)

3. Core Script — 외부 script 방식

- 그레고리가 core 로직을 외부 CDN(또는 그레고리 서버)에 호스팅한다
- 학생은 자기 웹사이트의 HTML에 script 태그 한 줄만 삽입하면 된다
- core script는 DOM을 일절 조작하지 않는다 (순수 데이터/네트워크 레이어)
- core script가 자동으로 처리하는 것 (학생이 신경 쓸 필요 없음):
    - URL 파라미터 파싱 (닉네임, 찬/반 여부)
    - architect ID 검증 (data attribute로 지정)
    - firebase 연동 (체류시간 heartbeat, 참여 데이터 전송)
    - 토론 상태 감시: firebase에서 현재 토론 상태를 실시간 조회
        - "진행중"이 아니면 savePayload를 비활성화하고 heartbeat를 중단
        - 콜백으로 status를 넘겨서 학생이 "토론이 종료되었습니다" 등 UI 처리 가능
    - Firestore 보안 규칙으로도 이중 차단: "진행중"이 아닌 토론에는 서버 레벨에서 write 거부 (core script 우회 방지)
- core script가 콜백으로 학생에게 넘겨주는 정보:
    - nickname: 접속한 학생 닉네임 (없으면 null)
    - side: 해당 학생의 찬/반 여부 ("pro" | "con")
    - role: 해당 토론에서의 역할 ("participant" | "architect" | "agendasetter")
    - status: 현재 토론 상태 ("pending" | "active" | "reviewing" | "closed")
    - schedule: 일자별 아젠다세터, 아키텍트
    - savePayload(data): custom 데이터를 firebase로 저장하는 함수 (토론이 active일 때만 동작)
- 학생이 콜백 데이터를 기반으로 직접 UI를 분기해야 하는 것:
    - nickname이 null이면 → "플랫폼을 통해 다시 접속하세요" 등 안내 표시
    - role이 "architect" 또는 "agendasetter"이면 → read-only UI 처리
    - status가 "active"가 아니면 → 토론 종료/대기 안내 표시
- 그레고리가 core를 업데이트하면 학생 프로젝트 수정 없이 즉시 반영됨

- 삽입 예시:
    ```html
    <script
        src="https://gregory-debate-core.example.com/debate-core.js"
        data-architect-id="본인닉네임"
    ></script>
    ```

- JS에서 사용 예시:
    ```js
    // core script가 로드되면 전역 객체로 접근 가능
    window.DebateCore.onReady(({ nickname, side, role, schedule, savePayload }) => {
        // nickname: 접속한 학생 닉네임 (없으면 null)
        // side: "pro" | "con"
        // role: "participant" | "architect" | "agendasetter"
        // schedule: 일자별 아젠다세터/아키텍트 정보
        // savePayload(data): custom 데이터를 firebase로 저장

        if (!nickname) {
            // 닉네임 없음 → 접속 차단 UI 표시
            return
        }
        if (role !== "participant") {
            // 아키텍트 또는 아젠다 세터 → read-only UI 표시
            return
        }
        // 여기서부터 토론 UI 구성
    })
    ```

4. 기록 데이터

- 체류시간 (heartbeat 방식) — core script가 자동 처리
- 학생별 참여 y/n 정보 .csv로 복사 가능
- 이것들은 그레고리의 firebase로 전송됨
- 학생별 참여 데이터
    - json blob방식으로 custom_payload 필드를 두어, 아키텍트가 자유롭게 데이터 구조를 설계할 수 있게 한다
    - 예: 투표 선택지, 추가 의견, 반응 등 아키텍트가 설계한 인터랙션 데이터를 자유 형식으로 저장
    - core script가 제공하는 savePayload 함수를 통해 저장

5. 금지 사항

- core script 태그를 제거하거나 src를 변경하는 행위
- URL 파라미터를 강제로 조작하는 스크립트를 삽입하는 행위는 부정행위로 간주
- core script를 우회하여 직접 그레고리의 firebase에 데이터를 전송하는 행위

6. 허용 사항

- UI/UX 전반 자유 설계 (레이아웃, 스타일, 인터랙션, 프레임워크 선택)
- custom_payload 데이터 구조 자유 설계
- 그레고리 firebase 이외에 추가 DB 솔루션 사용 가능 (단, 필수 데이터는 core script를 통해 반드시 그레고리 firebase로 전송)
- 추가 페이지, 컴포넌트, 라이브러리 자유 추가

7. Workflow

A. 그레고리 — 사전 준비 (3/31 이전)

    1. 토론 플랫폼 개발 및 배포 (리스트 뷰, 토론 아이템, 28일 테이블, Cloud Functions)
    2. core script 개발 및 CDN 배포
        - heartbeat, URL 파라미터 파싱, firebase 연동, ID 검증 등
    3. 기본 템플릿 개발 및 GitHub 업로드
        - core script 연동이 완료된 동작하는 토론 웹사이트
    4. 25명 학생 스케줄 랜덤 배정 (캘린더에 등록)
    5. 학생들에게 배포
        - 기본 템플릿 GitHub 레포 주소 (또는 파일)
        - core script 삽입 가이드 (README)
        - 토론 플랫폼 URL + 개인 닉네임/학번

B. 그레고리 — 매일 반복 (D-1, 전날 오후 6시 전)

    1. 아젠다 세터로부터 주제 DM 수신
    2. 아키텍트로부터 URL DM 수신
    3. 아키텍트 URL 접속 테스트
        - core script 정상 로드 여부 확인
        - data-architect-id가 올바른지 확인
        - heartbeat가 firebase에 찍히는지 확인
    4. 28일 테이블에서 해당 날짜 행에 데이터 입력 (url, title)
    5. (URL 접속 불가 시) 기본 템플릿 URL로 대체, 해당 아키텍트 0점 통보

C. 그레고리 — 매일 반복 (D-day, 오후 6시)

    1. 토론이 자동으로 "진행중"으로 전환되는지 확인
    2. 전날 토론이 "집계중"으로 전환되는지 확인

D. 그레고리 — 매일 반복 (D+1, 집계 중)

    1. 전전날 토론이 "종료"로 전환되는지 확인
    2. 종료된 토론의 코멘트 리스트 검토
    3. 베스트 인사이트 선정 + 선정 이유 작성
    4. 28일 테이블 상세 모달에서 미참여자, 결과 미입력자 확인
    5. (필요시) 체류시간 데이터 수동 입력

E. 그레고리 — 수시

    1. core script 버그 수정/업데이트 → CDN 재배포 (학생 프로젝트 영향 없음)
    2. 스케줄 변경 요청 처리 → 28일 테이블에서 수정
    3. 통계 데이터 csv 다운로드

---

F. 학생(아키텍트) — 자기 차례 전

    1. 기본 템플릿을 받는다 (GitHub clone 또는 파일 다운로드)
    2. data-architect-id를 본인 닉네임으로 변경
    3. 자기 호스팅에 배포 (Vercel, Netlify 등)
    4. 정상 동작 확인
        - core script 로드 확인 (콘솔에서 window.DebateCore 객체)
        - URL 파라미터로 닉네임, 찬/반이 정상 전달되는지 확인
        - heartbeat가 동작하는지 확인
    5. UI를 자유롭게 수정/설계
        - 템플릿을 부분 수정해도 되고, 완전히 새로 만들어도 됨
        - 인터페이스 설계에 대한 가설 수립
    6. 토론 시작 전날 오후 6시까지 그레고리에게 DM 제출
        - 배포된 URL + 인터페이스 설계 가설

G. 학생(아젠다 세터) — 자기 차례 전

    1. 토론 주제 선정 (찬/반이 명확히 갈리는 가치판단 주제)
    2. 주제 선정에 대한 가설 수립
    3. 토론 시작 전날 오후 6시까지 그레고리에게 DM 제출
        - 주제 + 주제 선정 가설

H. 학생(참여자) — 매일

    1. 토론 플랫폼 접속 (닉네임+학번)
    2. 진행중인 토론 클릭 → 찬/반 선택 (최초 1회, 수정 불가)
    3. 아키텍트의 토론 웹사이트로 이동하여 토론 참여
    4. (다음 날, 집계중) 전날 토론에 대한 코멘트 작성
        - 참여자: 설득 여부 y/n + 토론분석 (500자 이내)
        - 아젠다 세터/아키텍트: 가설 + 토론분석 (500자 이내)
    5. 코멘트는 집계 마감(오후 6시) 전까지 수정 가능
